Proof of Erasure Protocol

The cryptographic foundation of Lethe's verified destruction market.

Core Principle

The signature implies erasure. A valid ECDSA signature from a TEE-bound broker enclave key over (auditId, codeHash, "ERASED") constitutes cryptographic proof that the code was received, processed, audited, and destroyed.

Protocol Steps

1. Key Generation

Broker generates ECDSA key inside TEE enclave. Public key registered in BrokerRegistry.sol. Private key never leaves the enclave.

2. Audit Binding

Client submits code. keccak256(code) is computed, then LetheEscrow.createAudit(codeHash) creates an on-chain record. The auditId links to code hash immutably.

3. TEE Execution

Broker enclave receives code, runs static analysis, generates audit report, then performs NIST 800-88 erasure (3-pass: 0x00, 0xFF, random).

4. PoE Signing

Enclave signs keccak256(abi.encodePacked(auditId, codeHash, "ERASED")) with enclave private key. The ECDSA signature (r, s, v) is the PoE token.

5. On-chain Verification

LetheEscrow.resolve() recovers signer from signature + message. If signer matches registered broker, escrow is released. Trustless verification.

Security Properties

  • Non-repudiation — Broker cannot deny a valid PoE signature
  • Confidentiality — Code never leaves the TEE enclave in plaintext
  • Verifiable destruction — NIST 800-88 multi-pass ensures data is unrecoverable
  • Trustless — Anyone can verify PoE on-chain without trusting the broker
  • Attestation — TEE remote attestation proves the enclave ran unmodified code

zkPoE v2

For constant-time on-chain verification, Lethe implements a Groth16 zero-knowledge circuit that wraps the PoE verification. This allows O(1) gas cost verification regardless of audit complexity. The circuit proves knowledge of a valid PoE signature without revealing the signature itself.

Components

Client

Browser or application that submits code via HTTPS and receives real-time updates via WebSocket.

Broker (FastAPI)

Central orchestration service. Receives audit requests, manages lifecycle, coordinates with TEE enclaves. Exposes REST API and WebSocket endpoints.

TEE Enclave (Gramine/SGX)

Trusted Execution Environment where code analysis occurs. Currently runs as Docker simulation; production would use actual SGX or SEV hardware. Provides attestation quotes and sealed encryption keys.

Semgrep Scanner

Static analysis engine running inside the TEE enclave. Scans for vulnerabilities using pattern matching and rule evaluation.

Bridge Service

Monitors on-chain events (AuditCreated), triggers broker, submits PoE to escrow for resolution.

Base Sepolia Contracts

LetheEscrow manages deposits and verifies PoE signatures. BrokerRegistry maintains broker public keys and attestation status. Chain ID: 84532.

Data Flow

1. Submit — Client sends code to POST /audit/request. Broker stores request, returns audit_id.

2. Audit — Broker forwards code to TEE enclave. Semgrep runs inside enclave, generates findings.

3. Erase — TEE performs NIST 800-88 compliant destruction (multi-pass zeroing).

4. Prove — TEE signs keccak256(audit_id, code_hash, "ERASED") with enclave key. This signature is the Proof of Erasure.

5. Settle — Bridge submits PoE to LetheEscrow.resolve(). On-chain contract recovers signer, verifies against registered broker, releases escrow.

Vision Architecture

Open for Discussion — Help us build this

This is where Lethe is going. None of this is implemented yet. We are publishing this architecture openly so that security researchers, protocol engineers, and agent builders can challenge, refine, and contribute to these designs.

Discuss these proposals on GitHub Discussions.

1. Decentralized Performer Model

The current PoC uses a single centralized broker. The target architecture replaces this with a competitive market of independent performers, each running their own TEE environment.

Current:   Request → Broker (central) → Broker's TEE → Result

Target:    Request → Market (matching) → Performer's TEE → On-chain verification
                                           ↑
                                     Any TEE provider:
                                     • Self-hosted SGX
                                     • Phala Network (TDX)
                                     • Marlin Oyster
                                     • Oasis ROFL

Performers register by submitting a TEE attestation quote to BrokerRegistry. The market matches requests to performers via sealed-bid auction. Performers compete on price, speed, and reputation — not on privileged access.

2. Sealed-Bid Auction

When a requester submits an audit intent, qualified performers submit encrypted bids. After a designated window (e.g. 30 seconds), bids are revealed simultaneously. The market selects based on a combination of price, performer reputation, and specialization match.

This prevents front-running and ensures fair price discovery. The mechanism is not yet designed in detail — we welcome proposals.

3. Three-Tier LLM Verification Pipeline

The current PoC uses Semgrep for static analysis. The vision adds a layered verification architecture where untrusted input never reaches a privileged LLM directly.

Tier 1: Gatekeeper (no tools)
  Input:  Raw vulnerability report (untrusted)
  Output: Sanitized VLS (Vulnerability Logic Summary)
  Rule:   Strip injection attacks, structure into safe JSON

Tier 2: Analyst (sandboxed tools)
  Input:  Sanitized VLS only
  Tools:  execute_poc(), fetch_source(), run_static_analysis()
  Env:    TEE or sandbox — no persistent storage, no network
  Output: Verification report (verified / debunked / inconclusive)

Tier 3: Judge (no tools)
  Input:  Analyst report + VLS + metadata
  Output: accept / reject / request_more_info
  Rule:   confidence ≥ 0.8 → auto-settle

The key principle: a tool-bearing LLM never touches untrusted input directly. This prevents prompt injection from compromising the verification pipeline.

4. Asymmetric Reputation System

Reputation must make betrayal structurally unprofitable. The proposed formula:

Success:  S_new = S_old + (0.10 × (100 - S_old))       // slow growth
Failure:  S_new = S_old - (0.25 × S_old × 1.6^streak)  // exponential pain

Loss/gain ratio: 2.5×
Consecutive failure: exponential penalty
  1 failure:  -25%
  2 failures: -40%
  3 failures: -64%

Circuit breaker:
  Score ≥ 50: ACTIVE
  Score 30-49: WARNING (reduced privileges)
  Score < 30: SUSPENDED (blocked from market)

Reputation is bound to a Soulbound Token (SBT) — non-transferable. A performer's track record stays with them. Forking the market doesn't fork the reputation.

5. Dispute Escalation

Three levels, with increasing cost borne by the losing party:

Level 1 — Self-Verification

Requester verifies findings independently. Cost: free. Resolves most disputes.

Level 2 — Single Arbiter

A neutral arbiter agent reviews the case inside a fresh TEE. Cost: 5% of transaction value, paid by loser.

Level 3 — Three-Judge Panel

Majority vote from three independent arbiter agents. Cost: 15% of transaction value, paid by loser. Final and binding.

6. Domain Expansion

The escrow layer is designed to be domain-agnostic. The verification logic is a plugin. Expanding to new domains means writing a new verification plugin — not modifying the market infrastructure.

Phase 1: Smart contract security audit (current)
Phase 2: General code review
Phase 3: Data integrity verification
Phase 4: AI model security (adversarial robustness)

Lower layer (domain-agnostic):
  Escrow, matching, settlement, reputation

Upper layer (pluggable):
  Verification protocol per domain

7. TEE Limitations — Honest Assessment

TEE hardware has been breached before (Foreshadow 2018, Plundervolt 2019, SGAxe 2020, AEPIC Leak 2022). Lethe's core premise — "information exists only inside the TEE" — is only as strong as the TEE itself.

Mitigations on the roadmap:

  • Multi-vendor TEE — Mix Intel SGX + AMD SEV + ARM TrustZone. A single vendor vulnerability doesn't compromise the entire market.
  • ZK transition — Long-term, replace hardware trust with mathematical proof. Currently impractical for LLM inference cost, but the field is advancing rapidly.
  • Continuous self-audit — Use the market's own agents to audit the market's TEE infrastructure.

We believe TEE-based erasure is the best available primitive today. It is not perfect. We say this openly because security communities rightly distrust projects that claim invulnerability.

These designs are open proposals. Challenge them.

API Reference

The Broker exposes a REST API and WebSocket endpoint for audit lifecycle management.

Base URL

http://localhost:8000 (local) or your deployed broker endpoint.

Endpoints

POST /audit/request

Submit code for audit. Returns an audit_id for tracking.

Body:

{
  "code": "def bad(): password = '123456'",
  "language": "python"
}

Response: {"audit_id": "aud-xxxx", "status": "pending"}

GET /audit/status/{audit_id}

Check audit status. Returns current state, findings (if complete), and PoE data.

States: pending → running → completed → erased

WS /ws/audit/{audit_id}

WebSocket for real-time audit progress. Receives status updates as the audit progresses through each stage.

POST /audit/trigger

Manually trigger audit processing for a pending audit. Used by the bridge when on-chain events are detected.

GET /health

Health check. Returns broker status and available TEE backends.

GET /backends

List registered TEE backends and their attestation status (SGX, TDX, Mock).

Quick Start

# Start broker
cd broker && uvicorn main:app --port 8000

# Submit audit
curl -X POST http://localhost:8000/audit/request \
  -H "Content-Type: application/json" \
  -d '{"code": "def bad(): password = \"123456\"", "language": "python"}'

# Check status
curl http://localhost:8000/audit/status/aud-xxxx
